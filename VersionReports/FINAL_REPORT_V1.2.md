# StoryHaven AI — V1.2 Release Audit Report

V1.2's centerpiece is LoRA training (Modal GPU integration, built this release), plus a marathon of live bug-fixing driven by real usage against the running app — several of these were caught mid-incident, not via a planned audit pass. Findings below are grouped by how they were found, not by severity, since a lot of this release was reactive debugging of a brand-new subsystem rather than a scheduled review.

## Fixed in this pass

- **LoRA training: uncaught provisioning-phase exception.** `_execute_training_job`'s provisioning `try/except` only caught specific exception types (`ModalProvisionError`, `ModalNotConfigured`, `RuntimeError`) — an `httpx.ReadError` from a dropped upload connection escaped the whole async task uncaught. asyncio only logged "Task exception was never retrieved"; the job's DB status stayed stuck at `provisioning` forever with no visible failure anywhere in the app. Added a catch-all `except Exception`, mirroring the training loop's own, that logs and fails the job visibly.
- **LoRA training: custom HTTP upload/download was slow and could hang forever.** The original `ensure_model_cached`/`download_output` implementation (hand-rolled httpx streaming to bespoke `upload_model`/`download_output` FastAPI endpoints) measured under 15MB/s on a 2Gbps link and, once, hung indefinitely with zero error after Modal killed the receiving container mid-transfer (the connection never got a clean close signal). Rewritten to use Modal's own Python SDK (`modal.Volume.batch_upload`/`read_file`/`iterdir`) directly — confirmed ~440MB/s on the same file, same link, with proper error propagation instead of a silent hang.
- **LoRA training: a genuine multi-hour silent hang inside the training subprocess.** A real run sat completely frozen at a mid-training step — zero stdout, zero exception — for the entire 3-hour Modal function timeout before being force-killed, burning the full GPU cost for nothing recoverable beyond the last scheduled checkpoint. Added stall-detection (`_STALL_LIMIT_SETUP`/`_STALL_LIMIT_MID_TRAINING` in `modal_app/lora_train.py`) that kills the subprocess itself after 20 minutes of silence pre-training or 10 minutes mid-training, instead of waiting on the outer timeout. Root cause of the hang itself was never conclusively identified (a stray animated GIF in the training set was the leading suspect, removed, but the hang recurred on a subsequent run with the GIF gone and single-threaded data loading forced — most likely GPU/host-level infrastructure jitter on shared rented hardware, not this app's code).
- **LoRA training: orphan-job cleanup didn't cover every in-flight status.** `fail_stuck_jobs()` (the startup routine that marks interrupted jobs failed after a server restart) only checked `("queued", "provisioning", "training")` — missing `loading_base_model` and `saving`. A job whose task died during either of those phases (e.g. an in-session backend code edit triggering `uvicorn --reload`, which kills any in-flight asyncio task) was left stuck showing a stale status forever. Now covers all five real in-flight statuses.
- **LoRA training: model-cache upload zip-detection could silently destroy a real download.** The curl command generated for approved model requests initially used `file`'s text output to detect a zip bundle (unreliable — reported a real zip as plain "data" on one system), then a magic-byte fix still didn't distinguish a genuine PyTorch-serialized checkpoint (itself a zip container internally, members named `data.pkl`/`data/<n>`/`version`) from a naive multi-file zip bundle — the first fix's "any zip = extract and pick matching-extension files" logic found zero matches for a real torch checkpoint and silently discarded the entire 64MB download with no error. Fixed by checking for `data.pkl` inside the zip to tell the two cases apart.
- **Character generation: `dialogue` field could come back array-shaped instead of plain text.** The model sometimes returned the `dialogue` field as a JSON array of lines (or a string containing array-literal syntax) instead of the newline-joined `{{user}}:`/`{{char}}:` text the editor's field expects, despite prompt instructions. Fixed both the prompt (explicit about the required plain-string shape) and the parsing (`_normalize_dialogue` flattens either shape defensively).
- **Character generation: a malformed request body could 422 with a meaningless error.** `POST /characters/generate-from-description` used a strict Pydantic body model that could throw a generic "Input should be a valid dictionary or object to extract fields from" if the request body ever arrived shaped unexpectedly — reproduced intermittently in real browser use, never via direct API testing, root cause not conclusively identified. Rewritten to parse the raw body manually and degrade to the existing, actually-informative "description is required" for any non-dict shape instead of 422ing.
- **RBAC: hardcoded username checks replaced.** Four separate `if username == "zukaarimoto"` checks across the codebase replaced with a proper `users.role` column (`'user'|'admin'|'dev'`), gated by a `get_dev` dependency; only an existing Dev can grant/revoke Dev on another admin, closing a self-escalation path a single hardcoded check couldn't express anyway.
- **Model request lifecycle: "Done" mislabeled 26 real installs as "Rejected".** The admin panel's "Done" button reused the `/reject` endpoint, since there was no distinct "installed" status. Added `POST .../complete` and a genuine `implemented` status; repaired the 26 affected historical rows via the app's own corrected endpoint (not raw SQL).
- **Zoom/pan modal: backdrop-click-closes falsely triggered by a drag.** A pan drag ending outside the image (over the modal backdrop) produced a synthetic click targeting the backdrop itself, which `openModal`'s own "click outside closes" handler then treated as a real dismiss-click. Fixed with a one-time, capture-phase click-suppressor armed only when an actual drag (not a tap) occurred.
- **Zoom/pan: missing on 3 of 10 call sites, and mobile pan didn't work at all.** `_wireZoomPan` was wired into most image-viewing modals but missed the standalone Generate page's own zoom button, the Train LoRA training-images thumbnail zoom, and the Test LoRA result zoom. Separately, the pan-drag implementation was mouse-only (`mousedown`/`mousemove`/`mouseup`) — tap-to-zoom worked on touch devices (taps synthesize a click) but dragging to pan a zoomed image did nothing on mobile. Both fixed: all zoom modals now wired consistently, and touch equivalents (`touchstart`/`touchmove`/`touchend`) added alongside the mouse handlers.
- **Admin previews: missing per-category filter/refresh, and inconsistent button sizing.** Added a filter box (name/description/tag match) and a one-click refresh per model-preview category (checkpoints/LoRAs/upscalers/samplers/schedulers); a `.tool` icon button rendered at 0×0 in one panel context due to a missing scoped CSS rule, and a `justify-content:space-between` container spread two buttons apart once a second one was added — both fixed.
- **Category filter tabs standardized into one dropdown.** Three near-duplicate implementations of a row-of-category-tabs (checkpoint picker, LoRA picker, Creations catalog) replaced with one shared filter-icon-dropdown widget (`_categoryFilterHTML`/`_wireCategoryFilter`), each gaining a genuine "All" option they didn't have before (previously always forced picking one specific category).
- **The non-functional "Flux V2" category tab removed** from the model picker — it was UI-only with no real generation workflow behind it, and existed only as an unusable, confusing option.
- **"Personae" branding fully renamed to "storyhavenai"** — logger name, default log-file name, ComfyUI output-filename prefixes, the card-export/import extension key (backward-compatible: reads both the old and new key so already-exported cards still round-trip their category/hidden/appearance-tags metadata), and a `localStorage` key for custom chat styles (same backward-compat read-fallback for existing users' saved data). Deliberately left untouched: `legacy/personae.db` and its one reference in `modules/py/migrate_to_postgres.py` — a real legacy SQLite filename still on disk, not a branding string.
- **Tags field converted to a pill/chip input** — type a tag, comma or Enter commits it as a removable pill, replacing a plain comma-separated text box; the underlying save format (still a comma-joined string) is unchanged, so no backend/schema change was needed.
- **"Images" renamed to "Creations"** (nav label, page title, icon) ahead of planned video-generation support landing alongside image generation. The `/images` route/URL itself is unchanged, so existing links/bookmarks still work.

## Audited, confirmed correct — no fix needed

- **Modal cost/queue mechanics**: the training queue (`_training_queue`), abort's three-part immediate cleanup (`_aborted_jobs`, `_queue_leave`, `task.cancel()`), and per-job cost estimation all behave as designed under live testing against a real Modal dashboard.
- **Resume-from-LoRA step accounting**: verified the total steps declared across a chain of resumed runs sums correctly to the originally-intended total (no double-counting, no silently-dropped steps) — the only caveat is optimizer momentum resetting on each resume (not a bug, an inherent limitation of `--network_weights` not restoring full training state).
- **Admin-only LoRA visibility gating**: `lora_repo.gate_visibility` correctly hides every self-trained LoRA from regular users until an admin explicitly publishes it; verified directly against the database for three real trained checkpoints from this session, all still gated as expected.

## Known backlog (left untouched — low severity, or genuinely unresolved)

- **The root cause of the multi-hour training-subprocess hang was never conclusively identified.** Stall-detection now bounds the damage to ~10-20 minutes instead of 3 hours, but the underlying freeze (no exception, no stdout, recurring across different runs even after removing a suspected bad file and forcing single-threaded data loading) is still unexplained — most likely infrastructure-level jitter on Modal's shared rented GPU hosts, outside this app's visibility.
- **`GET /admin/logs`'s underlying JSONL ring buffer was not migrated** when the default filename changed from `personae.logs.jsonl` to `storyhavenai.logs.jsonl` — a deployment with an existing log history starts a fresh file rather than continuing the old one. Low impact (logs are already time-pruned at 24h) but worth noting for anyone relying on the old filename in external tooling.
- **Page-load chattiness** — every page load re-queries ComfyUI's `/object_info` live for checkpoints/LoRAs/samplers/schedulers with no caching, and the app's one-file-per-concern convention means ~40 separate static asset requests per load. Neither is a bug (the first is correctness — ComfyUI is the source of truth for installed models; the second is the deliberate no-build-step architecture) but both are real, measurable latency contributors flagged during this session and not addressed — a short-TTL cache on the ComfyUI option lists would be the highest-leverage fix if this becomes a priority.

## Second pass — multi-agent re-audit

A follow-up audit ran four parallel domain agents (security/secrets, LoRA/Modal pipeline, frontend JS regressions, backend/DB correctness) against the state of the repo after everything above shipped.

### Critical — found outside the audit, not by it

**Real account credentials and a log file were committed to the public GitHub repo.** `CLAUDE.md` (which has a "User account" section listing literal admin/test login credentials) and `.claude/settings.json` had been tracked in git since the very first commit; a log file (`personae.logs.jsonl`, ~2000 lines of operational log entries — job IDs, usernames, status changes, no secrets/passwords itself) was committed once during this release. This was not caught by either audit pass — it surfaced when Google Search Console flagged the live site for "deceptive pages," which led to discovering the tracked files while investigating. Remediated: both files stripped from the working tree and `.gitignore`'d, then purged from **all** git history via `git filter-repo` + force-push, verified with a fresh clone showing zero trace in any commit. Because `CLAUDE.md` existed since the first commit, purging it necessarily rewrote every commit hash including V1/V1.1 (the log file, by contrast, only ever existed in the V1.2 commit, so purging it alone would not have touched earlier history). **The credentials themselves must still be treated as compromised and rotated** — a history purge stops future visibility, it does not undo whatever was already visible on a public repo before the purge.

Lesson for future sessions: `.gitignore` entries only stop *new* files from being tracked; they do nothing for a file already committed before the ignore rule existed. Any file containing real credentials needs an explicit `git ls-files`/`git log --all -- <path>` check the first time it's noticed, not an assumption that adding it to `.gitignore` retroactively protects it.

### Fixed from this pass

- **SSRF: DNS-pin resolution failure failed open, not closed** (`backend/ssrf.py`, `resolve_pinned_host`). If `socket.getaddrinfo` errored at pin-time (after the hostname already passed its initial validation), the function logged a warning and returned the URL **unpinned** — silently reopening the exact DNS-rebinding race window the pinning exists to close. Now raises and rejects the request instead of proceeding unpinned.
- **Swallowed exception with no log, violating the project's own logging rule** (`backend/routers/characters.py`, `generate_from_description`). The manual body-parsing fix from the previous pass caught `request.json()` failures into a bare `except Exception: raw = None` with zero logging — any malformed/non-JSON request body now silently produced a generic "description is required" with no trace of what actually went wrong. Added `log.warning` with the real exception.
- **Category filter picker showed a stale/inconsistent selection on reopen** (`static/js/imagegen-picker.js`, both `openModelPickerModal` and `openLoraPickerModal`). If "Hide legacy" was already on and the last-picked category was a legacy-only one (SDXL/IL/Pony), reopening the picker kept filtering by that now-hidden category — the dropdown offered no way to see or deselect it, and the grid silently showed a restricted/near-empty result with no visible indication why. Now resets to "All" on open in that situation, matching the existing live-toggle behavior.

### Confirmed solid (both passes)

Auth (cookie flags, RBAC, login/registration/reset throttling, no self-escalation path), encryption-at-rest coverage across every sensitive field, Modal secrets never logged/returned, the model-request host-allowlist never auto-fetching arbitrary URLs, custom-HTML sandboxing (sandboxed iframe with no `allow-scripts` + DOMPurify), the training queue/abort/cancellation lifecycle, resume-from-LoRA path-traversal safety (`os.path.basename` correctly applied), every exception path in `_execute_training_job` now terminating the job visibly, `transfer_progress`/metrics bounding, frontend poller/listener lifecycle, `db.py` migrations for the two new LoRA-job columns, the `storyhavenai` rename's internal consistency (no stray write path still using the old card-extension key), `fail_stuck_jobs()` now covering every real in-flight status, and the tags-pill input's save-path correctness (empty tags, commas-in-tags, hidden-field sync).

### New backlog (deferred, low/medium severity)

- **Stall-detection can false-trigger during a legitimately slow, silent checkpoint save** (`modal_app/lora_train.py`) — the only stall-reset signal is subprocess stdout, so a multi-GB `.safetensors` write on a slow disk with zero chatter for >600s gets killed as a false stall. Worth either exempting the save window or emitting a periodic heartbeat line during it.
- **Reverse-proxy `X-Forwarded-Proto` trust unverified** (`backend/auth.py`'s session cookie `Secure` flag depends on `request.url.scheme`) — if uvicorn isn't configured to trust the proxy's forwarded-proto header, a session cookie could theoretically be issued non-secure behind a TLS-terminating proxy. Not independently confirmed broken, flagged for a follow-up check of `run.sh`'s uvicorn flags.
- **`modal_provision.py` logs/surfaces up to 4000 chars of raw third-party `modal deploy` output verbatim** — no secret leak found (Modal CLI doesn't echo the token env vars), but unbounded third-party CLI output surfaced raw is worth a defensive scan/redaction pass as hardening.
- **`_normalize_dialogue`'s nested-list edge case** (`backend/ai_helpers.py`) — a deeply-nested list value (rare, malformed-source-data-only) gets stringified as a Python repr rather than recursively flattened. Cosmetic, not a crash.
- **Shared `hideLegacy` localStorage key across the checkpoint and LoRA pickers** (`ig_mp_hide_legacy` used by both, no separate `ig_lp_hide_legacy`) — likely intentional as one global preference, inconsistent with the per-picker category-key naming right next to it; not confirmed as unintended.
- **`_wireZoomPan`'s window-level pointer listeners self-heal rather than never accumulate** — rapid open/close of a zoom modal with zero intervening pointer movement can pile up stale listener sets until the next real mouse/touch event anywhere triggers their cleanup. Self-resolves on first interaction; not a true unbounded leak, but not the "never accumulates" guarantee its own comment claims either.
- **Queued-status LoRA jobs show zero dedicated panel** (`static/js/lora-training.js`) — the phase-gating logic's stated invariant is "exactly one panel visible," but `status==="queued"` matches none of the five gates, so only the plain status line shows. Possibly intentional (nothing to display yet); flagged as a deviation from the documented invariant either way.
- **`flux_v2` residual asymmetry** — removed from the filterable category tabs, but still assignable to a checkpoint via the admin edit-meta modal and still handled by `imagegen.js`'s generation-blocking logic. Not broken, just inconsistent (classifiable but not filterable).

## Process note

Unlike V1.1's five-parallel-domain-agent audit, most of V1.2's fixes were discovered live: a real admin actually using the newly-built LoRA training feature against a real Modal account, hitting real stalls/crashes/cost, in real time. Several "fixes" in this report are second or third attempts at the same underlying problem (the zip-detection bug was patched twice; the training stall was addressed after two different runs died two different ways) — this is reflected honestly above rather than presented as clean one-shot fixes. The second-pass multi-agent audit caught real, previously-unflagged issues (the SSRF fail-open, the stale category filter), but notably **missed** the actual most serious incident of the release (credentials committed to a public repo) — that was found by accident via an external Google flag, not by either audit. Worth remembering next release: a dedicated "check git history for anything that shouldn't be tracked" pass is not automatically covered by a code-content-focused security audit.
