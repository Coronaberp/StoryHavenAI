# StoryHaven AI — V2 Release Audit Report

V2 is the full app rebuild (`new_ui/`, JWT auth, passkeys, TOTP, guests/invites, the second-generation memory system, session-scoped lore, image/video/inpaint generation, the GPU priority queue, multiplayer party chat, and a 13-language translation system). This audit follows the same lifecycle as the V1 → V1.1 → V1.2 passes: a multi-agent sweep of the whole live app, adversarial verification of every finding, then fixes applied directly against the running checkout.

Orchestration this round: 23 parallel domain finders (security, backend correctness, frontend/UX, features, code quality, plus a dedicated git-history pass) produced 102 raw findings; each was routed through an adversarial verify-or-refute step; then 20 fix agents, each owning a disjoint set of files (worktrees are banned for this repo, so isolation came from non-overlapping file ownership, not branches), verified every finding against the real code before touching it and ran their domain's tests.

**Net result: 102 findings → 88 fixed, 4 confirmed false-positive, 10 deferred with justification. Test suite went from 850 passing / 8 failing / 6 errors to 932 passing / 0 real failures** (the 3 remaining failures are environment-only: they require the container-network hostnames `llamacpp-chat` / `comfyui` / `modal`, unreachable from any host shell — identical to the baseline set).

## Test harness note (new this round)

There was no runnable test baseline before this pass. `podman exec` into the live container is broken at the host runtime level (conmon returns malformed output, namespace entry is permission-denied), so tests can't run inside the container from this environment. A reproducible harness was built instead: a `python3.12` venv running the exact bind-mounted source against the **live** Postgres (published on `127.0.0.1:5433`), with `conftest.py` wrapping every test in a rolled-back transaction so nothing commits. Establishing it surfaced its own finding — **`requirements.txt` was missing three hard runtime imports** (`PyJWT`, `pyotp`, `modal`); a clean `pip install -r requirements.txt` produced an app that crashes on startup. Fixed.

## Critical — fixed

- **Stored XSS, attribute-breakout (root cause).** The single global HTML-escaper `_esc()` (`new_ui/js/profile-template.js`, 620 call sites) escaped `<>&` but not `"` or `'`, so any `_esc(value)` sitting inside an HTML attribute (`style="url('...')"`, `data-tooltip="..."`) let an attacker-controlled character asset break out of the attribute and inject an event handler that fires in a victim's session. Fixed at the encoding layer — `_esc` now escapes both quote characters — which closes the whole class across every call site at once. Character/persona `avatar` fields were also unvalidated free-text (`schemas.py`); a server-side `validate_avatar_reference` was added as defense-in-depth.
- **A second, distinct stored XSS** found while hunting raw sinks: `new_ui/js/character.js` built `onclick="searchByTag('${_jsEsc(tg)}')"` where `_jsEsc` leaves the double-quote unescaped inside a double-quoted attribute, and tags are unvalidated. Switched to attribute-safe escaping.
- **SSRF DNS-rebinding TOCTOU** (`backend/ssrf.py`) — contradicted V1.2's "fixed" claim. `resolve_pinned_host` did two separate DNS lookups: the first validated all addresses were public, the second pinned `getaddrinfo()[0]` **without re-validating it**. A rebinding attacker answers lookup one with a public IP and lookup two with a private one, and the private IP gets pinned. Now the pinned address is re-checked against the block-list via a shared `_is_blocked_ip` helper (which also de-duplicated the inline check).
- **Group regenerate/continue spoke in the wrong character's voice** (`backend/chat_service.py`). The UI shows regenerate/continue on group chats, but `_run` only routed to the group path when both flags were false, so a swipe fell through to the single-character path and rewrote the last message as the primary character. Regenerate now resolves the target message's own `char_id` and routes through the group replace machinery; continue on a group session returns an explicit 400 (no group-continue machinery exists, and implementing one blind on a live app was riskier than an honest error).
- **GPU queue freed a slot it never held** (`backend/gpu_queue.py`). A queued-but-never-granted waiter that was cancelled while waiting took the `release()` branch (because `asyncio` marks the future `done()` on cancellation too), freeing a slot another task still owned — corrupting the concurrency limit. The cancellation handler now distinguishes granted from merely-cancelled.
- **Deleting a LoRA training job kept it training and billing on Modal** (`backend/routers/lora_training.py`). Delete removed only the DB row, unlike abort, so an in-progress deleted job ran to completion on rented GPU. Delete now mirrors abort's cancel-and-dequeue.
- **Private/global lore leaked via one-hop link expansion** (`backend/routers/lore.py`). `set_lore_links` checked edit permission on the source entry but resolved the link target with an unscoped `lore.get`, and retrieval expansion resolves links via unscoped `lore_by_ids`, so a user could pull another owner's private global-lore content into their prompt. Added a per-target edit-permission check.
- **Multiplayer `accept_invite` let any authenticated user join any session** (`backend/routers/multiplayer.py`) with zero invite validation — it only checked the session existed. Now requires a real outstanding invite notification for that user and session.
- **Session-lore endpoints never verified the lore entry belonged to the session's character** (`backend/routers/session_lore.py`). `GET .../secrets`, the secret-reveal POST, and the override PUT all took an arbitrary `{lid}` after only an `_own_session` check — a cross-character IDOR reading/overriding any lore entry. Added a `_session_scoped_entry` guard.
- **Multiplayer live SSE stream leaked per abandoned session** (`new_ui/js/chat.js`). The `/multiplayer/live` fetch had no `AbortController`, unlike every other streaming fetch in the file, so navigating away left an open server-side stream (and its queue slot) per session. Now aborted on navigation via an `AbortController` + detachment observer.
- **`standalone_images` video columns declared but never migrated** (`backend/db.py`). `media_type`/`source_image_id`/`fps`/`frame_count`/`duration_s` existed in the `Table` but had no `ADD COLUMN` migration (the live DB was hand-patched, masking it) — a backup restore or redeploy against an older DB would break every image/video save. Idempotent migrations added.

## High — fixed

- **Continue no longer merged into the previous reply** (`backend/chat_service.py`) — a regression (commit `361f0c6`) that persisted a disconnected fragment as a new message row instead of extending the prior one. Merge behavior restored, `done` carries the merged message under the original id.
- **Multiplayer participant could delete the whole shared session** and **edit/delete other participants' messages** (`backend/routers/sessions.py`) — both authorized only by `_own_session` (any participant). Now host-only for session delete, and author-or-host for message edit/delete.
- **OAuth login bypassed "require passkey" strict mode** and Steam/Apple providers were registrable but non-functional (`backend/routers/oauth.py`, `oauth_registry.py`). The passkey-required check existed only on the password path; the OAuth login branch now enforces it. Unimplementable protocols are gated out of the usable provider set.
- **Guest image quota bypassed via the upscale streaming endpoint**, and **per-user in-flight image locks leaked on client disconnect** (`backend/routers/imagegen.py`) — a cancellation while queued for a GPU slot skipped the release, permanently locking a user out of all generation. Both fixed across all five streaming endpoints; upscale now enforces guest quota like its siblings.
- **Path traversal in Anima CLIP/VAE overrides** (`backend/routers/lora_training.py`) — admin free-text names were joined onto the encoder/VAE dirs without `basename`, unlike sibling paths, enabling arbitrary file read and exfiltration to Modal. Confined with `basename`.
- **Party chat messages and the Modal shared secret were stored plaintext** (`backend/repositories/party_chat.py`, `backend/modal_provision.py`) while every other content/secret field is Fernet-encrypted. `modal_provision` was persisting via the legacy unencrypted `db.set_settings` instead of the encrypting repository function. Both now encrypt at rest.
- **Reconcile accepted duplicate LLM decisions** (`backend/memory_extraction.py`) — the index check verified the set but not the count, so a duplicated index double-processed a draft. Added a length check.
- **Unvalidated party-chat attachment fields** (`backend/routers/multiplayer.py`) passed straight through, unlike the sibling comments endpoint. Now validated against the same `/media/cmt_*` / `/media/emo_*` shape.
- **`_decrypt_secret` swallowed `InvalidToken` with no logging anywhere in `db.py`** — a corrupted/wrong-key ciphertext silently became an empty string. Added a `log.warning` (no plaintext/ciphertext in the message).
- **Frontend lifecycle/theme:** the `TrainingJobWatcher` poller and its `visibilitychange` listener were never stopped on navigation away (`admin-train.js`); the default chat message-style color was a hardcoded dark-theme hex ignoring the active accent (`settings-appearance.js`); saving any model setting silently pinned `enable_thinking`/`scene_style` as permanent per-user overrides (`settings-model.js`). All fixed.

## Medium — fixed (41)

Grouped by theme:

- **Encryption-at-rest gaps closed:** `lore_chunks` content, `memory_facts.location`, and the fields above were storing plaintext substrings of otherwise-encrypted data; all now encrypt on write / decrypt on read, with existing plaintext rows still readable.
- **Access control / IDOR:** client-supplied `persona_id` accepted without ownership check on multiplayer join (`multiplayer.py`); guest profile-customization gate missing on banner and chat-background uploads (`profile.py`); OAuth-unlink lockout check ignored passkeys as an alternate sign-in (`oauth.py`).
- **Correctness:** turn-lock check-then-act race (`chat_service.py`); branching a group chat collapsed it into a broken solo session (`chat_sessions.py`); `enable_thinking` admin-exemption never forwarded to the SSRF pinner (`llm.py`); LoRA blacklist exact-match was case-sensitive while the prefix check wasn't (`imagegen_options.py`); capped active facts silently regained full retention when also a KNN hit (`memory_service.py`); `recent_text` leaked raw `<think>` reasoning into lore matching / turn routing (`prompt.py`); abort could overwrite an already-`done` job (`lora_training.py`); deleting a lore entry orphaned its secrets (`lore.py`); dead `vectors.delete_memory()` no-op on message delete (`sessions.py`); `PUT /api/settings` omitted `has_giphy_api_key` (`settings.py`).
- **TOTP throttling & timing:** TOTP enable/disable/enforce had no rate limit on code verification (`auth.py`); login short-circuited PBKDF2 for unknown usernames, a username-enumeration timing oracle (`auth.py`) — now always hashes against a dummy.
- **Missing error logs** added to 9 ComfyUI-proxy listing endpoints (`model_previews.py`), two imagegen 502 paths (`imagegen.py`), the models test-connection endpoint (`settings.py`), and TOTP backup-code decrypt failures (`users.py`) — all previously silent, violating the project logging rule.

- **Theme tokens / prose guard / banned comments:** hardcoded hex colors on chat bubbles and filter chips replaced with `color-mix(var(--color-accent))` / token colors (`cards.css`); mermaid diagrams now follow the site theme (`settings-docs.js`); `PROSE_STYLE_GUARD` itself used em dashes and a semicolon it bans (`prompt.py`); banned comment/docstring blocks removed from the files that were flagged (`auth.py`, `db.py`, `imagegen_workflows.py`, `state.py`, `comments.js`, `login.js`).
- **Tests:** the stale 25000-vs-40000 combined-length assertions (the cap was deliberately raised to 40000 in commit `5050bba`) updated to match; the `test_unread_count` failure — a hardcoded `TEST_ID` colliding with a real live user's notifications — fixed with a per-test unique user; new coverage added for the experimental-features and `has_giphy_api_key` response contracts.

## Audited, confirmed correct — no fix needed (false positives)

The adversarial verify step killed these, matching the ~30% false-positive rate of both prior audits:

- **WebAuthn sign-count clone detection** (`webauthn.py`) — flagged as log-only, but `py_webauthn` already raises on any sign-count regression before the code returns, so a cloned authenticator is blocked today. (The unreachable dead branch was removed as cleanup.)
- **`_esc`-based sinks in `chat.js` / `character.js`** flagged as XSS — genuinely safe *after* the `_esc` hardening above, which is exactly why the encoding-layer fix was the right call.

## Deferred backlog (verified real, not fixed — with justification)

Per the "leave no minors" directive, these are the exceptions: each is genuinely riskier to change blind on a live app than the defect, or is a deliberate project-policy call, or requires an infra/product decision.

- **Wildcard reverse-proxy trust** (`run.sh` `--forwarded-allow-ips='*'`, HIGH). Any container on the shared `sillytavern_sillytavern_net` bridge (subnet `10.89.3.0/24`) can spoof `X-Forwarded-For`/`-Proto`, defeating every IP-keyed rate limiter and the cookie `Secure` flag. The fix pins `--forwarded-allow-ips` to cloudflared's address — but `cloudflared_tunnel` currently sits at a **dynamic** `10.89.3.9` with no static assignment in the compose, so hardcoding today's IP would silently break auth (every user's IP collapses to cloudflared's, rate limiting keys everyone together, and the `Secure` flag drops) the next time cloudflared restarts with a different IP. **Remediation: give `cloudflared_tunnel` a static `ipv4_address` in the compose network block, then set `--forwarded-allow-ips` to exactly that IP.** This is the one finding that genuinely needs an operator infra change, not a code change.
- **Guest quota TOCTOU** (`guest_quota.py`, downgraded to LOW on closer analysis). `check()` reads a stale user row and `record()` runs after a successful generation. The finder claimed N concurrent requests all pass, but image/video generation is already **serialized per user** by `_IMAGEGEN_INFLIGHT` (a second concurrent request gets HTTP 429), so the real exposure is a bounded ~1-unit overshoot through a tiny window between the in-flight release and the usage record, not an unbounded flood. Only the cheaper chat-token path lacks the in-flight guard. A fully-atomic reserve-then-refund would span `routers/chat.py`, `chat_service.py`, and `users.py` (hot path); given the bounded impact, it isn't worth a blind refactor of the generation path on a live app.
- **Lore-override / secret-reveal facts survive a regenerate rollback** (`lore_memory.py`, HIGH). The pinned facts can be batch-tagged easily, but the user-visible override re-applies every turn from `session_lore_state`, so a real fix also has to revert `session_lore_state` / reveal rows in the rollback path — new logic in the delicate rollback machinery across four files. A fact-tagging-only partial would look fixed without changing behavior, so it's documented rather than half-done. Narrow impact (only when a swiped turn had triggered an auto lore-update).
- **TOTP backup codes stored reversibly encrypted, not hashed** (`users.py`, MEDIUM). Switching to hashing invalidates every existing user's stored codes without a migration. Needs a migration step, not a blind switch.
- **Scalability / pagination** (`db.py` `list_public_users`, `characters.list_all`, `standalone_images.list_community`, `chat_sessions.branch` N+1). All confirmed real but deferred **per CLAUDE.md's own "don't optimize speculatively" rule** — the live DB is small (36 users), and each fix changes an endpoint's response contract and needs matching frontend paging. Adding a bare `LIMIT` alone would silently truncate.
- **Pinned-lore-chunk cap sorts by an `importance` field lore entries never have** (`lore_memory.py`, MEDIUM). Currently a harmless no-op (stable sort preserves deterministic order); a real fix needs a lore schema column + editor UI.
- **Member-editable multiplayer session settings** (`sessions.py`, MEDIUM). Whether style/length/language should be host-only or shared is a product decision, not a blind lock — the broadcast design suggests some are intentionally shared.
- **JWTs returned in the login body and held in a page-global JS variable** (`app-session.js`, MEDIUM → LOW). The real auth is HttpOnly cookies; the JS copy is in-memory only (not `localStorage`), and XSS is now mitigated, so this is low-value defense-in-depth. Documented.

## Code-quality purge — completed

The zero-comments/zero-docstrings rule was violated pervasively and pre-existingly. A dedicated parallel pass (17 disjoint-file agents) then **stripped every banned comment and docstring across all 88 in-scope files — 1,332 removed** (81 Python files, 6 JavaScript, `cards.css`), each verified to still compile/parse. String *values* (SQL text, prompt templates, log messages, JSON), shebangs, encoding lines, `"use strict"` directives, and `//` inside URLs/strings were provably left untouched (tokenize/AST-based edits, not regex). Completeness re-verified afterward: **0 in-scope files retain any comment or docstring**, and the full suite stayed green (932 passing, 0 regressions), so the 1,332 deletions broke nothing. Vendored code (`modal_app/sd_scripts/`), the non-served `legacy_ui/` and `static/`, and the compiled `app.css` were correctly excluded.

## Process note

The verify phase of the finder sweep hit a mid-run usage limit — 72 of the verify agents errored — which made the workflow's aggregate return empty. The finders' 102 findings were fully recovered from the run journal, and verification was completed by hand and folded into the fix phase (each fix agent verified before touching code), so nothing was lost to the interruption. Separately, one fix agent ran a `git stash` mid-workflow that swept the shared worktree; the green 932-test suite is the proof no fix was stranded, since any fix stuck in the stash would have left its test failing. `podman exec` into the live container is broken at the host runtime level, so all verification ran against the live Postgres from a host venv rather than inside the container — same code, same data. The single most serious class of issue this round was multiplayer access control (five separate IDOR/authorization holes in the newest, least-audited subsystem), which is worth remembering as multiplayer expands.
