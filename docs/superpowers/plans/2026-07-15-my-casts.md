# My Casts Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `PantheonView` (`new_ui/js/pantheon.js`) configurable by scope and reuse it directly for `/sanctum/casts` (the user's own character library), replacing that route's current placeholder — no new view file.

**Architecture:** `PantheonView` gains a constructor option (`{ scope: "community" | "mine" }`) that drives its fetch scope, disables creator-filtering UI/logic when scoped to `"mine"`, hides the card's creator row, and swaps header/error copy. `characterCardHtml` gains a third `opts` parameter to support the hidden-creator-row card variant. Both `/pantheon` and `/sanctum/casts` instantiate the same class with different constructor args.

**Tech Stack:** Vanilla JS classes + Tailwind utility classes (frontend, no build step, matching every other `new_ui/` screen).

## Global Constraints

- Zero comments in any file, ever — no exceptions (per `CLAUDE.md` coding style).
- No hardcoded hex colors outside `themes.css` — this plan doesn't add new CSS, but any markup touched must keep using existing `var(--color-*)`-backed classes, never introduce a literal hex.
- `new_ui/` has no JS test runner — verification is manual against the human's already-running `./rebuild.sh --watch` dev server on `:3001`, per `CLAUDE.md`. Never spin up a second dev server instance for this.
- Never use `EnterWorktree`/`git worktree` for this repo — edit `/var/home/staygold/ai-frontend` directly (bind-mounted into the live container).
- This is a live, shared checkout — other agents may be editing the same files concurrently. Re-read a file immediately before editing it if there's any chance it changed since last read. Commit only the files this task actually touches.
- This task edits `pantheon.js`, which already powers the live `/pantheon` route — every change must be verified as a **non-regression** on `/pantheon` (creator filter, creator card row, community-scoped fetch all still work exactly as before), not just as a new feature on `/sanctum/casts`.

- [ ] **Step 4: `load()` — scope-driven fetch, skip creator profile loading when scoped to `"mine"`, scope-aware error**

Change:

```javascript
  async load() {
    this.loading = true;
    this.error = "";
    this.render();
    try {
      const params = new URLSearchParams({ scope: "community" });
      if (this.filters.q) params.set("q", this.filters.q);
      if (this.filters.tags.length) params.set("tags", this.filters.tags.join(","));
      this.chars = await api(`/api/characters?${params.toString()}`);
    } catch (err) {
      this.error = err.message || "Couldn't load the Pantheon.";
      this.chars = [];
    }
    this.loading = false;
    this.render();
    this.loadCreatorProfiles();
  }
```

to:

```javascript
  async load() {
    this.loading = true;
    this.error = "";
    this.render();
    try {
      const params = new URLSearchParams({ scope: this.scope });
      if (this.filters.q) params.set("q", this.filters.q);
      if (this.filters.tags.length) params.set("tags", this.filters.tags.join(","));
      this.chars = await api(`/api/characters?${params.toString()}`);
    } catch (err) {
      this.error = err.message || (this.scope === "mine" ? "Couldn't load your characters." : "Couldn't load the Pantheon.");
      this.chars = [];
    }
    this.loading = false;
    this.render();
    if (this.scope === "community") this.loadCreatorProfiles();
  }
```
